## 5. 数据流测试

### 5.1 运行时数据流量分析

当前系统运行时的数据流主要分为四类：

1. 文本输入流  
   用户导入 TXT 文本后，文本被拆分为章节和段落，写入 `books` 状态并存入 IndexedDB。

2. 文本分析流  
   阅读器将“当前段落 + 上下文 + 角色地点摘要”发送给文本模型。  
   数据特点：
   - 请求内容主要是纯文本与结构化 JSON
   - 单次数据量通常远小于图片流量
   - 对浏览器内存压力较小

3. 图片生成流  
   图片模型返回远程 URL，而不是直接回传大二进制图像到前端状态。  
   随后本地中间件根据 URL 发起一次下载，把图片写入 `pic_db/`。

4. 本地持久化流  
   业务状态和图片记录写入 IndexedDB；图片原图写入仓库本地目录。

### 5.2 详细数据流转图

```mermaid
flowchart TD
    A[TXT 文本] --> B[createBook]
    B --> C[books state]
    C --> D[IndexedDB app_state]

    E[阅读器段落] --> F[analyzeNarrative]
    F --> G[NarrativeFacts]
    G --> H[generateIllustration]
    H --> I[远程 URL]
    I --> J[/api/save-generated-image]
    J --> K[pic_db/]
    K --> L[illustrations state]
    L --> M[IndexedDB image_records]
    L --> N[页面 img 展示]
```

## 6. 性能测试与分析

### 6.1 构建性能

实测构建命令：

```bash
npm run build
```

实测结果：
- 构建时间：约 1.24 s
- `dist/index.html`：1.85 kB
- `dist/assets/index-BTOM9wDi.js`：311.77 kB
- Gzip 后 JS：94.46 kB

说明：
- 当前前端体量仍处于轻量级区间
- 单包结构尚可接受，但 `App.tsx` 与 `Reader.tsx` 业务集中度较高，后续继续增长时建议拆分模块

### 6.2 本地图片存储规模

实测 `pic_db/` 当前状态：
- 文件数：13
- 目录体积：约 12 MB

分析：
- 当前一张图片平均约 0.9 MB 左右
- 若按长篇小说累计生成数百张图，本地磁盘占用会快速上升
- IndexedDB 只记录元信息，不直接保存全部图片二进制，降低了浏览器存储压力

![地点生成测试样例](../pic_db/book-white-bone/assets/locations/loc-tiger-ridge-1775720667795.jpg)

### 6.2.1 运行时数据流量组成分析

从链路上看，系统流量主要由两部分构成：

1. 文本分析流量  
   主要是 JSON 和纯文本，体量较小，通常只包含：
   - 当前段落文本
   - 上下文文本
   - 角色/地点摘要
   - 风格参数

2. 图片生成与图片回填流量  
   体量明显更大，包含：
   - 图片模型请求
   - 生成后远程图片下载
   - 启动补档时的历史图片回填下载

因此，本系统运行时的数据流量瓶颈主要在图片链路，而不是文本链路。

### 6.3 单段生图性能分析

单段生图的主要耗时分布：
1. 文本分析请求
2. 图片生成请求
3. 本地图片下载与写盘
4. React 状态更新与页面重渲染

其中：
- 最大耗时通常在“图片生成请求”
- 本地写盘是新增步骤，但它把远程临时 URL 转换为了可长期使用的本地文件，属于可接受开销
- 页面渲染压力主要来自大图 `<img>` 加载，不在文本分析阶段

建议补充观测指标：
- 点击生图到“进入生成中”的前端响应时间
- 图片模型返回 URL 的耗时
- 本地落盘耗时
- 图片出现在页面上的总耗时

### 6.4 批量生图性能分析

当前批量生图采用并发队列，而不是完全串行。

并发策略：
- 默认批量并发数：3
- 先分析段落
- 再批量补齐缺失角色
- 再并发生成插图

优点：
- 相比串行模式，显著减少总等待时间
- 相比完全并发，避免短时间内堆积过多请求导致 UI 卡顿或 API 失败率升高

风险点：
- 同时保存多张图片时，`pic_db/` 写盘会有瞬时 I/O 峰值
- 浏览器短时间加载多张大图时，内存占用会上升

建议的压测方式：
- 选择包含 20 段以上文本的章节
- 批量策略选 `all`
- 记录以下指标：
  - 总任务数
  - 成功数 / 失败数
  - 平均单张耗时
  - 最大并发时 UI 是否卡顿
  - `pic_db/` 在批量执行前后的体积变化

### 6.5 运行时资源消耗分析

| 资源 | 主要消耗点 | 结论 |
|---|---|---|
| CPU | React 渲染、图片解码、批量任务调度 | 正常场景可接受 |
| 内存 | 多张图片同时展示、任务队列状态 | 批量生成阶段较敏感 |
| 网络 | AI 文本分析与图片生成、补档下载 | 图片生成阶段为主 |
| 磁盘 | `pic_db/` 累积存图 | 长期使用需关注容量增长 |

### 6.6 启动恢复性能分析

当前应用启动除了加载 React 页面，还会做两类恢复工作：
- 从 IndexedDB 恢复业务状态
- 对历史图片执行本地补档检查

这意味着：
- 当图片记录较少时，启动恢复成本较低
- 当历史图片很多且存在缺失文件时，启动阶段会有额外网络和 I/O 开销

测试建议：
- 分别在“0 张图、10 张图、100 张图”三级数据规模下测量首页可交互时间
- 观察补档逻辑是否导致页面长时间停留在加载态

## 7. 已发现的性能瓶颈

1. `App.tsx` 负责状态过多  
   任何核心状态变化都会触发较大范围的持久化与渲染判断。

2. 图片文件持续增长  
   当前策略保留原图，长期使用下 `pic_db/` 增长较快。

3. 批量阶段缺少细粒度资源监控  
   虽然已有阶段标签，但尚未做请求耗时、失败率、磁盘写入时间统计。

4. 导出 PDF 仍依赖浏览器打印  
   对大批量图片页面的导出稳定性有限。

5. 启动补档会放大首屏恢复成本  
   历史数据规模大时，启动阶段的检查和回填可能变慢。

6. 图片记录与业务状态仍然同步保存  
   当前设计已经拆出 `image_records`，但主状态依然会整体持久化，写入频率偏高。

## 8. 优化建议

### 8.1 近期可做

- 为设置页增加“图片记录数量 / 本地图片总量”可视化
- 为批量任务增加耗时统计
- 给 `pic_db/` 增加清理工具
- 将 `Reader.tsx` 里的任务引擎进一步模块化
- 为启动补档增加进度条和节流策略
- 为异常场景补统一错误码和埋点

### 8.2 中期可做

- 增加图片缩略图缓存策略
- 对大图做懒加载
- 将图片记录和业务状态拆分成更细的存储粒度
- 引入更稳定的 PDF 导出方案
- 增加自动化 UI 测试与性能基线
